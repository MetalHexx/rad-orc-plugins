# Action & Event Reference

Quick-lookup tables for the Orchestrator. See [pipeline-guide.md](pipeline-guide.md) for the full event loop, CLI usage, error handling, and operational context.

> **ALWAYS FOLLOW THE GATE PROTOCOL in state.json** When `ask` or `never` is  selected for any gate listed in the projects state.json.  No exceptions, even in Copilots "autopilot", Claude Codes "Auto Mode" or any other similar YOLO mode.  The human must be the one to approve or reject at every gate when `ask` or `never` is selected.  This is non-negotiable and critical to the proper operation of this system.  Do not attempt to bypass or automate human approval at gates under any circumstances.

## Action Routing Table

Every `data.action` value from the pipeline envelope maps to exactly one Orchestrator operation. All branching derives from this table.

| # | `data.action` | Category | Orchestrator Operation | Event to Signal on Completion |
|---|---------------|----------|----------------------|-------------------------------|
| 1 | `spawn_requirements` | Agent spawn | **Two-step protocol:** (1) Signal `requirements_started` with `{}` context â†’ pipeline returns `spawn_requirements` again; (2) Spawn **planner** agent in Requirements mode. The agent authors the lean FR/NFR/AD/DD ledger that master_plan and later tasks reference by ID. Output: {NAME}-REQUIREMENTS.md. Before spawning, read `repository_skills_block` from the envelope's `data.context` block and inline it verbatim into the spawn prompt per the orchestrator agent's *Planner Spawn Manifest* section (omit entirely when the field is the empty string). | `requirements_completed --doc-path <output-path>` |
| 2 | `spawn_master_plan` | Agent spawn | **Two-step protocol:** (1) Signal `master_plan_started` with `{}` context â†’ pipeline returns `spawn_master_plan` again; (2) Spawn **planner** agent. The agent reads the project's Requirements doc and performs targeted codebase discovery (Grep/Glob/Read) to produce the inlined phase + task plan. Output: {NAME}-MASTER-PLAN.md. Before spawning, read `repository_skills_block` from the envelope's `data.context` block and inline it verbatim into the spawn prompt per the orchestrator agent's *Planner Spawn Manifest* section (omit entirely when the field is the empty string). Also inline a `## Plan Size Limits` block listing `max_phases` and `max_tasks_per_phase` from `data.context.limits` per the orchestrator agent's *Planner Spawn â€” Plan Size Limits* section (heading string is contractual; do not paraphrase). | `master_plan_completed --doc-path <output-path>` |
| 2a | `explode_master_plan` | Script (no agent) | **Deterministic script**, no agent spawn. (1) Signal `explosion_started` with `{}` context â†’ pipeline returns `explode_master_plan` again; (2) Invoke the explosion subcommand `node "${COPILOT_VSCODE_PLUGIN_ROOT}/skills/rad-orchestration/scripts/radorch.mjs" plan explode` with `--project-dir`, `--master-plan`, `--project-name`. The subcommand parses the approved Master Plan, emits per-phase + per-task docs into `phases/` + `tasks/`, and pre-seeds `state.graph.nodes.phase_loop.iterations[]` â€” each phase iteration carries `doc_path` directly (relative path to the phase plan), and each task iteration carries `doc_path` directly (relative path to the task handoff). No synthetic child step nodes are seeded; body nodes are scaffolded by the walker on each iteration's first in_progress transition. Downstream consumers (`execute_task`, `spawn_phase_reviewer`, etc.) read these `doc_path` values directly â€” no per-phase or per-task authoring action fires at loop time. **Re-run**: if `phases/` or `tasks/` already contain anything, the script first moves the entire contents to `backups/{ISO-timestamp}/`, then re-seeds. **Exit codes**: `0` success â†’ signal `explosion_completed`; `2` for both parse failures and real failures (the CLI framework maps any `ok: false` envelope to exit 2) â€” disambiguate by envelope shape: `ok: true` with `data.error` present â†’ parse failure â†’ signal `explosion_failed` with the structured error envelope: extract `{ line, expected, found, message }` from `data.error`, format as a JSON string, and pass it to the parse-error flag; `ok: false` with `error.type: system_error` â†’ real failure â†’ route through `rad-log-error`. | `explosion_completed` (success) or `explosion_failed --parse-error '{"line":...,"expected":...,"found":...,"message":...}'` (parse failure) |
| 3 | `execute_task` | Agent spawn | **Two-step protocol:** (1) Signal `execution_started` with `{}` context â†’ pipeline returns `execute_task` again; (2) Spawn **coder** agent. Context carries `handoff_doc` as the sole doc-path input â€” the pre-seeded task handoff path (original or active corrective, via `context-enrichment.ts` routing) â€” plus phase/task identifiers (`phase_number`, `phase_id`, `task_number`, `task_id`). No Requirements / Master Plan / upstream docs. Output: source code + tests + optional `## Execution Notes` appendix to the handoff body. | `task_completed` |
| 4 | `spawn_code_reviewer` | Agent spawn | **Two-step protocol:** (1) Signal `code_review_started` with `{}` context â†’ pipeline returns `spawn_code_reviewer` again; (2) Spawn **reviewer** agent for task-level code review. Context includes `head_sha` â€” the commit hash of the task's commit (resolved from active corrective task when applicable, else task iteration). `head_sha` is `null` when `source_control.auto_commit: never` or no commit has been made; in that case the reviewer falls back to `git diff HEAD` + untracked files. Output: reports/{NAME}-CODE-REVIEW-P{NN}-T{NN}-{TITLE}.md. Orchestrator mediation on `verdict: changes_requested` sits between reviewer completion and event signaling (see `corrective-playbook.md`). | `code_review_completed --doc-path <output-path>` |
| 5 | `spawn_phase_reviewer` | Agent spawn | **Two-step protocol:** (1) Signal `phase_review_started` with `{}` context â†’ pipeline returns `spawn_phase_reviewer` again; (2) Spawn **reviewer** agent for phase-level review. Context includes `phase_first_sha` (first task's initial commit) and `phase_head_sha` (last task's latest commit, corrective-aware). Both SHAs are `null` when `source_control.auto_commit: never` or no commits have been made; in that case the reviewer falls back to `git diff HEAD` + untracked files. Reviewer runs `git diff <phase_first_sha>~1..<phase_head_sha>` to get the cumulative phase diff for the skeptical pass. The reviewer emits a single artifact that combines the conformance verdict with the structured phase summary (Task Results, Exit Criteria Assessment, Files Changed, Issues & Resolutions, Corrections Applied, Carry-Forward Items). Output: reports/{NAME}-PHASE-REVIEW-P{NN}-{TITLE}.md. Orchestrator mediation on `verdict: changes_requested` sits between reviewer completion and event signaling (see `corrective-playbook.md`). | `phase_review_completed --doc-path <output-path>` |
| 6 | `spawn_final_reviewer` | Agent spawn | **Two-step protocol:** (1) Signal `final_review_started` with `{}` context â†’ pipeline returns `spawn_final_reviewer` again; (2) Spawn **reviewer** agent for final comprehensive review. Context includes `project_base_sha` (first chronological commit across the project, derived from iteration commit_hash values) and `project_head_sha` (last committed SHA, including corrective commits at both task and phase scope). Both SHAs are `null` when `source_control.auto_commit: never` or no commits have been made; in that case the reviewer falls back to `git diff HEAD` + untracked files. Reviewer runs `git diff <project_base_sha>~1..<project_head_sha>` to get the cumulative project diff for the skeptical pass. Output: reports/{NAME}-FINAL-REVIEW.md. Iter-12: final-review corrective cycles are NOT wired â€” verdict is strict-and-final; no orchestrator mediation participates on this event. | `final_review_completed --doc-path <output-path>` |
| 7 | `request_plan_approval` | Human gate | Display Master Plan summary to the human. Ask human to approve or reject. | `plan_approved` (if approved) or `plan_rejected` (if rejected) â€” no context payload |
| 8 | `request_final_approval` | Human gate | Display final review to the human. Ask human to approve or request changes. **PR link:** If `pipeline.source_control.pr_url` is present in state, include a `PR: {pr_url}` line in the prompt so the reviewer can navigate directly to the pull request. If `pr_url` is absent, omit the PR line entirely â€” do not show an empty placeholder or error. | `final_approved` (if approved) or `final_rejected` (if rejected) â€” no context payload |
| 9 | `gate_task` | Human gate | Show task results to the human. Wait for approval. | `gate_approved --gate-type task` (if approved) or `gate_rejected --gate-type task --reason "<reason>"` (if rejected) |
| 10 | `gate_phase` | Human gate | Show phase results to the human. Wait for approval. | `gate_approved --gate-type phase` (if approved) or `gate_rejected --gate-type phase --reason "<reason>"` (if rejected) |
| 11 | `ask_gate_mode` | Human gate | Present the three gate mode options (`task`, `phase`, `autonomous`) to the operator. Wait for selection. | `gate_mode_set --gate-mode <chosen>` |
| 12 | `display_halted` | Terminal | Display `data.context.details` to the human. **Loop terminates.** | *(none â€” terminal action)* |
| 13 | `display_complete` | Terminal | Display completion summary to the human. **Loop terminates.** | *(none â€” terminal action)* |
| 14 | `invoke_source_control_commit` | Agent spawn | Spawn **source-control** in commit mode. Pass in the prompt: `worktree_path`, `task_id` (from `data.context`), task title and type (from the handoff). Extract `commitHash` and `pushed` from the agent's `## Commit Result` block. | `commit_completed --commit-hash <hash> --pushed <true|false> --phase <N> --task <N>` |
| 15 | `invoke_source_control_pr` | Agent spawn | Spawn **source-control** in PR mode. ## Event Signaling Reference

These are the exact event names passed to `--event`:

| Event | Flags (besides `--event` and `--project-dir`) | When to Signal |
|-------|-----------------------------------------------|----------------|
| `start` | *(none)* | First call (new project), cold start, or context compaction recovery |
| `requirements_started` | *(none)* | Before Planner spawn (Requirements mode). Transitions `graph.nodes.requirements.status` to `in_progress` and flips `graph.status` â†’ `in_progress` (this hook relocated from `master_plan_started` in Iter 4). See action #1 two-step protocol. |
| `requirements_completed` | `--doc-path <path>` | After Planner finishes the Requirements doc. Frontmatter must carry `requirement_count: <positive integer>`. |
| `master_plan_started` | *(none)* | Before Planner spawn (master plan). Transitions `graph.nodes.master_plan.status` to `in_progress`. See action #2 two-step protocol. |
| `master_plan_completed` | `--doc-path <path>` | After Planner finishes (master plan) |
| `explosion_started` | *(none)* | Before invoking the explosion script. Transitions `graph.nodes.explode_master_plan.status` to `in_progress`. See action #2a two-step protocol. |
| `explosion_completed` | *(none)* | After the explosion script finishes successfully. The script has already emitted per-phase + per-task docs and pre-seeded `phase_loop.iterations[]`. Clears any `master_plan.last_parse_error` / `parse_retry_count` recovery state. |
| `explosion_failed` | `--parse-error '<json>'` | After the explosion script exits with code 2 (parse failure). Payload is the structured error: `{ line, expected, found, message }`. The mutation handler stores the error on `master_plan.last_parse_error`, increments `master_plan.parse_retry_count`, and re-routes back to `@planner` for self-correction. **Retry cap**: hardcoded 3. The 4th consecutive failure halts the pipeline via `rad-log-error` ("Explosion parser rejected planner output N times (cap=3). Manual intervention required."). |
| `plan_approved` | *(none)* | After human approves master plan |
| `plan_rejected` | *(none)* | After human rejects master plan |
| `source_control_init` | `--branch <name> --base-branch <name> --worktree-path <path> --auto-commit <always\|yes\|never\|no> --auto-pr <always\|yes\|never\|no> [--remote-url <url>] [--compare-url <url>]` | After `rad-execute-parallel` creates the worktree (or at the start of `rad-execute` when `pipeline.source_control` is still null). One-time initialization that persists source control context to `pipeline.source_control` in state. `yes`/`no` are normalized to `always`/`never` respectively (case-insensitive); any other value (including `ask`, empty, or missing) is rejected with a descriptive error. Remote and compare URLs are optional; omitted or empty values are stored as `null`. |
| `execution_started` | *(none)* | Before Coder spawn. Transitions `task_executor.status` to `in_progress`. See action #3 two-step protocol. |
| `code_review_started` | *(none)* | Before Reviewer spawn (task-level). Transitions `code_review.status` to `in_progress`. See action #4 two-step protocol. |
| `task_completed` | `--doc-path <path>` *(optional, ignored)* | After Coder finishes task. The CLI accepts `--doc-path` for backward compatibility, but the pipeline ignores it. |
| `code_review_completed` | `--doc-path <path>` | After Orchestrator mediation finishes (or immediately after reviewer finishes when verdict is `approved` or `rejected`). **Frontmatter contract**: the reviewer always writes a raw `verdict` (`approved` \| `changes_requested` \| `rejected`). When the orchestrator mediates (raw `verdict: changes_requested`), it adds three fields to the review doc's existing frontmatter before signaling: `orchestrator_mediated: true`; `effective_outcome` â€” one of `approved` \| `changes_requested` (the outcome after applying per-finding judgments); and `corrective_handoff_path` â€” the relative path to the corrective Task Handoff, present if and only if `effective_outcome === 'changes_requested'`. When raw verdict is `approved` or `rejected`, none of these orchestrator fields appear. |
| `commit_started` | `[--phase <N>] [--task <N>]` | Signaled when the walker reaches the `commit` node in `task_loop.body`. `--phase` and `--task` are optional; auto-resolved from the active in-progress phase/task when omitted. |
| `commit_completed` | `--commit-hash <hash> --pushed <true\|false> [--phase <N>] [--task <N>]` | After Source Control Agent completes. Extract `commitHash` and `pushed` from the agent's `## Commit Result` JSON block. `--phase` and `--task` are optional; auto-resolved when omitted. |
| `pr_requested` | *(none)* | Signaled internally after `final_review_completed` when `auto_pr: always` and `pr_url` is **undefined** (absent from state â€” not yet attempted). A `null` value means PR creation was attempted but no URL is available; `null` does **not** re-trigger `pr_requested`. Validation checkpoint before Source Control Agent spawn in PR mode. |
| `pr_created` | `--pr-url <url>` *(optional)* | After Source Control Agent completes PR creation. Extract `pr_url` and `pr_number` from the agent's `## PR Result` JSON block. On success, signal with `--pr-url <url>`. On failure (`pr_url` is `null` in the result), signal `pr_created` **without** the `--pr-url` flag â€” the pipeline CLI will omit `pr_url` from context and the mutation handler will coalesce it to `null`. Writes `pr_url` to `state.pipeline.source_control`. |
| `phase_review_started` | *(none)* | Before Reviewer spawn (phase-level). Transitions `phase_review.status` to `in_progress`. See action #5 two-step protocol. |
| `phase_review_completed` | `--doc-path <path>` | After Orchestrator mediation finishes (or immediately after reviewer finishes when verdict is `approved` or `rejected`). **Frontmatter contract**: the reviewer always writes a raw `verdict` (`approved` \| `changes_requested` \| `rejected`) and `exit_criteria_met` (`true` \| `false`). When the orchestrator mediates (raw `verdict: changes_requested`), it adds three fields to the review doc's existing frontmatter before signaling: `orchestrator_mediated: true`; `effective_outcome` â€” one of `approved` \| `changes_requested` (the outcome after applying per-finding judgments); and `corrective_handoff_path` â€” the relative path to the corrective Task Handoff, present if and only if `effective_outcome === 'changes_requested'`. When raw verdict is `approved` or `rejected`, none of these orchestrator fields appear. The reviewer's raw `exit_criteria_met` is preserved across mediation and never overwritten. |
| `gate_mode_set` | `--gate-mode task\|phase\|autonomous` | After operator selects gate mode |
| `gate_approved` | `--gate-type task\|phase` | After human approves a gate |
| `gate_rejected` | `--gate-type task\|phase --reason <text>` | After human rejects a gate |
| `final_review_started` | *(none)* | Before Reviewer spawn (final review). Transitions `final_review.status` to `in_progress`. See action #6 two-step protocol. |
| `final_review_completed` | `--doc-path <path>` | After final reviewer finishes. **Frontmatter contract**: the reviewer writes a `verdict` field (`approved` \| `changes_requested` \| `rejected`), enforced by the pre-read frontmatter validator (iter-12). Final-review corrective cycles are out of scope â€” no mediation fields (`orchestrator_mediated`, `effective_outcome`, `corrective_handoff_path`) participate on this event. |
| `final_approved` | *(none)* | After human approves final review |
| `final_rejected` | *(none)* | After human rejects final review |
| `halt` | *(none)* | Emergency stop â€” signals the pipeline to halt immediately |

> [!NOTE]
> **Task auto-resolution:** For task-scoped events (`commit_started`, `commit_completed`), the `--phase` and `--task` flags are optional. When omitted, the engine resolves the active phase and task from state automatically. If no single unambiguous active phase or task exists (zero or multiple `in_progress`), the event fails with a descriptive error and instructs the operator to pass `--phase <N>` and/or `--task <N>` explicitly.

> [!NOTE]
> **Phase auto-resolution:** For phase-scoped `_started` events (`phase_review_started`), the `--phase` flag is optional. When omitted, the engine resolves the active phase from state automatically. If no single unambiguous active phase exists (zero or multiple phases `in_progress`), the event fails with a descriptive error and instructs the operator to pass `--phase <N>` explicitly.
